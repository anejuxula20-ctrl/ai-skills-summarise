## 方法与结论

本期以 GitHub REST API 的仓库快照为主（2026-06-21），再用仓库公开 README 与最近推送时间判断持续活跃度。GitHub API 不直接提供无认证的“过去 7 天新增 star”字段，因此本文不把无法复核的增星数字写成事实；“近期信号”指当日或近期推送、持续维护和所在主题的开发者关注度。

## 核心技术亮点

1. **代理正走向可执行闭环。** Codex、Gemini CLI 和 Goose 都把编辑、执行、测试等动作纳入代理工作流。可验证的完成条件比“更像人”更重要。
2. **工作流而非单轮对话成为产品载体。** Dify 代表的方向是将编排、部署、评估和运营合并，便于业务团队复用和审计。
3. **MCP 与浏览器扩大了可调用世界。** mcp-use 和 browser-use 表明连接外部系统正成为差异化能力，但也会让安全边界显著复杂化。

## 对开发者和产品团队的建议

- 先做狭窄、可测量的任务闭环：固定输入、工具集、完成条件和人工兜底，再逐步扩大自治范围。
- 对 MCP、浏览器和 shell 工具采用最小权限、短期凭据、可回放日志和按环境隔离的策略。
- 将成功率、人工接管率、工具错误率、引用正确率等指标纳入发布门槛，避免只看 demo 成功率。
- 选型时验证许可证、提交节奏、issue 响应、依赖树和文档完整性，而不是仅依据 star。

## 风险、争议与不确定性

- **star 不等于采用。** 发布、媒体曝光和营销都可能制造短期波动；应同时看维护和真实集成证据。
- **代理权限是主要风险面。** 能执行命令、控制浏览器或连接 MCP 服务的工具可能暴露文件、令牌和业务数据。
- **研究代理需要事实控制。** 引用缺失、过期网页与错误归纳会形成“看起来合理”的错误结论。
- **快照局限。** 本报告记录的是单日 GitHub API 数值；若需要严格 24h/7d 排序，应在自动化中持久化每日快照并计算差分。
